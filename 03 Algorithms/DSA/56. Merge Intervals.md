## Summary

Given an array of intervals `intervals` where `intervals[i] = [starti, endi]`, merge all overlapping intervals and return an array of the non-overlapping intervals that cover all the intervals in the input.

Constraints: `1 <= intervals.length <= 10^4`, `intervals[i].length == 2`, `0 <= starti <= endi <= 10^4`.

**Example 1:**
Input: `intervals = [[1,3],[2,6],[8,10],[15,18]]`
Output: `[[1,6],[8,10],[15,18]]`
Explanation: Since intervals `[1,3]` and `[2,6]` overlap, merge them into `[1,6]`.

**Example 2:**
Input: `intervals = [[1,4],[4,5]]`
Output: `[[1,5]]`
Explanation: Intervals `[1,4]` and `[4,5]` are considered overlapping (they touch at `4`).

## Hints

1. Overlap is hard to detect when intervals are in arbitrary order. What single operation would make overlaps trivial to spot as you scan left to right?
2. Sort the intervals by start value first. After sorting, any interval that overlaps the "current" merged interval must immediately follow it.
3. Walk through the sorted intervals keeping a running "current merged interval." For each next interval, compare its start against the current interval's end.
4. If `next.start <= current.end`, they overlap (or touch) — extend `current.end` to `max(current.end, next.end)`. Otherwise, the current interval is finished: push it to the result and start a new current interval.
5. Don't forget to push the final `current` interval after the loop ends.

> [!note]- Solution (click to reveal)
> **Approach:** Sort the intervals by start time. Then greedily sweep left to right, maintaining a "current" merged interval. For each subsequent interval, if its start is less than or equal to the current interval's end, they overlap (or touch), so extend the current interval's end to the max of the two ends. Otherwise the current interval can't grow anymore — commit it to the result and make the next interval the new "current." After the loop, commit whatever is left in "current."
>
> **Why this works:** Once sorted by start, the only interval that can possibly overlap the current merged interval is the very next one in the sorted order (any interval further along starts even later, so if it doesn't overlap the next one, it can't overlap the current one either). This greedy, single-pass merge is therefore always correct — no interval is ever "skipped over" incorrectly.
>
> **Complexity:** Time `O(n log n)` dominated by the sort (the merge sweep itself is `O(n)`). Space `O(n)` for the sorted copy and the output list (or `O(log n)` / `O(n)` extra for the sort's own working memory, depending on the sort implementation).
>
> ```java
> import java.util.*;
>
> class Solution {
>     public int[][] merge(int[][] intervals) {
>         if (intervals == null || intervals.length <= 1) {
>             return intervals;
>         }
>
>         // Sort by start time.
>         Arrays.sort(intervals, (a, b) -> Integer.compare(a[0], b[0]));
>
>         List<int[]> merged = new ArrayList<>();
>         int[] current = intervals[0];
>         merged.add(current);
>
>         for (int i = 1; i < intervals.length; i++) {
>             int[] next = intervals[i];
>             if (next[0] <= current[1]) {
>                 // Overlaps (or touches) the current interval — extend it.
>                 current[1] = Math.max(current[1], next[1]);
>             } else {
>                 // No overlap — start a new current interval.
>                 current = next;
>                 merged.add(current);
>             }
>         }
>
>         return merged.toArray(new int[merged.size()][]);
>     }
> }
> ```
