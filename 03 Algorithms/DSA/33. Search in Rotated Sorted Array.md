## Summary

There is an integer array `nums` sorted in ascending order (with **distinct** values). Prior to being passed to your function, `nums` is possibly rotated at an unknown pivot index `k` (`1 <= k < nums.length`) so that it becomes `[nums[k], nums[k+1], ..., nums[n-1], nums[0], nums[1], ..., nums[k-1]]` (0-indexed).

Given the (possibly rotated) array `nums` and an integer `target`, return the index of `target` if it exists in `nums`, or `-1` otherwise. The algorithm must run in **O(log n)** time.

**Constraints:** `1 <= nums.length <= 5000`; `-10^4 <= nums[i] <= 10^4`; all values in `nums` are unique; `nums` is an ascending array possibly rotated; `-10^4 <= target <= 10^4`.

**Example 1:** `nums = [4,5,6,7,0,1,2]`, `target = 0` → `4`

**Example 2:** `nums = [4,5,6,7,0,1,2]`, `target = 3` → `-1`

## Hints

1. Plain binary search needs a fully sorted array — but even after rotation, at least one of the two halves `[lo..mid]` / `[mid..hi]` is always properly sorted.
2. At each step, compare `nums[lo]` and `nums[mid]` to figure out which half is the sorted one.
3. Once you know the sorted half, check whether `target` falls inside its value range; if it does, recurse/iterate into that half, otherwise go into the other half.
4. Be careful with `<=` vs `<` at the boundaries (`nums[lo] <= nums[mid]`, and the range checks) — that's where off-by-one bugs hide.
5. You never need to explicitly locate the rotation pivot first; the half-sorted check handles it implicitly in one pass.

> [!note]- Solution (click to reveal)
> **Approach:** Run a modified binary search. At each step with window `[lo, hi]` and `mid = lo + (hi - lo) / 2`:
> - If `nums[mid] == target`, return `mid`.
> - Otherwise, determine which half is sorted by comparing `nums[lo]` and `nums[mid]`:
>   - If `nums[lo] <= nums[mid]`, the left half `[lo, mid]` is sorted. If `target` lies within `[nums[lo], nums[mid])`, search left (`hi = mid - 1`); otherwise search right (`lo = mid + 1`).
>   - Otherwise the right half `[mid, hi]` is sorted. If `target` lies within `(nums[mid], nums[hi]]`, search right (`lo = mid + 1`); otherwise search left (`hi = mid - 1`).
> - Repeat until `lo > hi`, returning `-1` if the loop exits without a match.
>
> **Why this works:** A rotated sorted array always has the property that splitting it at any `mid` leaves at least one contiguous half fully sorted. Knowing that half's value range lets us decide in O(1) whether `target` could be in it, so we discard half the search space each iteration exactly like classic binary search.
>
> **Complexity:** Time `O(log n)` — one binary search pass. Space `O(1)` — only a few index variables.
>
> ```java
> class Solution {
>     public int search(int[] nums, int target) {
>         int lo = 0, hi = nums.length - 1;
>
>         while (lo <= hi) {
>             int mid = lo + (hi - lo) / 2;
>
>             if (nums[mid] == target) {
>                 return mid;
>             }
>
>             if (nums[lo] <= nums[mid]) {
>                 // Left half [lo..mid] is sorted.
>                 if (nums[lo] <= target && target < nums[mid]) {
>                     hi = mid - 1;
>                 } else {
>                     lo = mid + 1;
>                 }
>             } else {
>                 // Right half [mid..hi] is sorted.
>                 if (nums[mid] < target && target <= nums[hi]) {
>                     lo = mid + 1;
>                 } else {
>                     hi = mid - 1;
>                 }
>             }
>         }
>
>         return -1;
>     }
> }
> ```
